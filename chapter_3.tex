The modernization of AnitoPlume involves migrating the rendering pipeline from a traditional to a modern graphics API architecture. 
This migration introduces a fundamentally different graphics architecture characterized by explicit resource management, reduced driver overhead, and improved utilization of modern multicore processors.
These architectural changes are beneficial for simulators such as AnitoPlume, which require continuous updates of thousands of dynamic particle billboards.

\section{Graphics Rendering Architectures}
Traditional graphics API architectures are designed to abstract low-level graphics operations from the programmer by implicitly managing graphics state, resource synchronization, and command validation within the graphics driver.
Throughout this study, these are referred to as implicit graphics APIs, whereas architectures that expose these responsibilities directly to the application are referred to as explicit graphics APIs.

\subsection{Overview of Graphics API Architectures}
Implicit graphics APIs such as OpenGL are based on a state-based programming model where the application modifies global rendering states that are maintained by the graphics implementation \cite{opengl46spec2022}.
Implicit graphics API architecture relies on a thick driver layer that abstracts graphics pipeline management from the application (Fig. 1).
Because the driver is responsible for maintaining graphics state, performing runtime validation, tracking resource usage, and managing command execution, significant CPU overhead is introduced during rendering (Ping et al., 2026).

Explicit graphics architectures expose synchronization, resource management, and command submission responsibilities directly to applications \cite{vulkan14spec2024}.
Rather than relying on implicit driver-managed state changes, explicit graphics architectures require the application to explicitly define and manage the configuration of the graphics pipeline.
Applications explicitly define pipeline states, resource allocation, synchronization, and memory management as seen on figure \ref{fig:api_architecture_comparison}.

\subsection{Limitations of the Implicit API Architecture}
The limitations of the traditional graphics API architecture become particularly significant in computationally intensive scientific visualization systems such as AnitoPlume. 
In addition to numerically integrating volcanic plume dynamics including gravity, buoyancy, drag, and wind forces for thousands of particles each simulation step, the CPU must also handle graphics driver validation and state management \cite{del2026interactive}. 
This added overhead limits rendering performance, motivating the use of lower-overhead graphics APIs to reduce CPU-side workload for applications that require high-throughput rendering. 
As rendering workloads become more complex, these driver-managed operations can limit the application's ability to optimize resource usage, creating CPU-side bottlenecks that reduce the rate at which work can be submitted to the GPU \cite{ping2026modern}.

\subsection{Performance Optimization opportunities}
Explicit graphics architectures reduce driver-managed serialization by shifting the responsibility to the application.
Tasks such as pipeline configuration, resource allocation, and command preparation can be performed explicitly by the application before or outside the main rendering loop. During runtime, the system only executes predefined instructions without modifying or accessing the underlying data.
This shift allows command generation to be safely distributed across multiple CPU threads \cite{ping2026modern}.
The lightweight driver architecture introduced by explicit graphics APIs minimizes runtime validation and hidden state management, reducing CPU overhead while providing developers with greater control over rendering operations. By enabling application-driven optimization and more efficient utilization of hardware resources, explicit graphics API architectures are better suited for performance-critical rendering workloads \cite{IoannidisC.2020MRFC}.

\section{Lagrangian Particle Model for Volumetric Plume Dynamics}
Volcanic eruption plumes are complex multiphase systems consisting of volcanic ash, gases, and entrained atmospheric air. 
Their evolution is governed by interactions between gravitational forces, buoyancy, atmospheric resistance, and environmental conditions, resulting in dynamic structures that change continuously over time. 
In computer graphics and scientific visualization, such phenomena can be represented using Eulerian and Lagrangian formulations (Esposti Ongaro et al., 2020).

This study adopts the Lagrangian plume formulation proposed by \citeA{lastic2022interactive} and the theoretical foundation for volcanic plume behavior in AnitoPlume (Del Gallego et al., 2026). 
Rather than introducing a new physical model, this research preserves the established plume dynamics formulation and focuses on the modernization of the underlying graphics architecture and the extension of simulator capabilities. 
The adopted formulation provides the mathematical basis for representing plume evolution, including particle motion, buoyancy effects, atmospheric interaction, and density-driven behavior.

The Lagrangian formulation represents the plume as a collection of independently evolving volumetric slices. 
Each slice maintains its own physical state variables and undergoes force evaluation, numerical integration, and property updates based on local parameters.
This particle-based structure exhibits data parallel characteristics because the computations applied to individual slices are largely independent.
Therefore, the model is suitable for parallel execution architectures where multiple particle states can be processed concurrently.

Therefore, the adopted plume model establishes the computational requirements that motivate the architectural migration explored in this study. 
Since the underlying physical formulation remains unchanged, improving simulator performance requires addressing the efficiency of the execution and rendering pipeline rather than reducing the complexity of the physical model.

\subsection{Governing Particle Dynamics}
The evolution of the volcanic plume is defined by the physical equations introduced by \citeA{lastic2022interactive}. 
The model represents each plume element as a particle governed by Newtonian mechanics, where its motion is determined by the combined influence of gravitational force, buoyancy, atmospheric friction, and wind forces.

Gravity provides the primary downward force acting on the particle due to its mass and density, while buoyancy produces upward acceleration caused by the density difference between the hot volcanic mixture and the surrounding atmosphere. 
Atmospheric friction reduces particle velocity through resistance, while wind forces introduce lateral movement that affects plume deformation and direction.

The particle state is updated through numerical integration over discrete time steps. 
In addition to external forces, the model incorporates air entrainment, which represents the continuous mixing between the volcanic mixture and surrounding atmosphere. 
This process modifies particle density and momentum over time, allowing the plume to expand and evolve naturally during ascent.

Through this formulation, plume behavior emerges from the collective interaction of individual particles rather than from predefined animation paths. 
This enables physically based visualization of eruption columns while maintaining a formulation suitable for interactive simulation.

\subsection{Computational Characteristics of Particle-Based Plume Simulation}
The adopted Lagrangian plume formulation represents a computationally intensive workload due to the continuous processing required for large numbers of dynamic particles. 
Each simulation step requires evaluation of physical forces, numerical integration of particle states, and preparation of updated data for rendering.

Unlike static graphical objects, plume particles are continuously changing in position, velocity, density, and appearance. 
As the number of active particles increases, the amount of computation and data movement required by the simulation increases accordingly. 
Additional physical behaviors, such as density evolution and regime transitions, introduce further computational requirements due to their dependence on particle-level calculations.

Furthermore, the simulation and rendering processes are closely coupled. 
The physical model generates updated particle states, while the rendering pipeline must continuously transform these states into visual representations such as particle billboards or volumetric approximations. 
This creates a high-throughput workload where simulation accuracy, rendering efficiency, and resource management must operate together.

Since this study retains the established Lagrangian plume formulation, these computational demands remain inherent to the system. 
Therefore, the primary challenge addressed by this research is not the simplification of plume physics, but the modernization of the execution architecture responsible for processing and rendering the particle-based simulation. 
This requirement provides the theoretical motivation for migrating from implicit graphics API architectures toward explicit graphics API architectures.
